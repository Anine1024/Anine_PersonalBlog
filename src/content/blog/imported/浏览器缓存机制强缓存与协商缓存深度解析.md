## 引言

在前端开发中，性能优化是一个永恒的话题。用户打开网页的速度直接影响用户体验和网站留存率。而浏览器缓存，作为性能优化中最简单高效的手段之一，能够显著减少网络请求、降低服务器负载、缩短页面加载时间。然而，许多开发者对缓存的理解仅停留在“使用缓存”的层面，对于强缓存和协商缓存的具体区别、配置方式以及实际应用场景却知之甚少。

本文将结合 HTTP 协议中的关键头部字段，深入剖析浏览器缓存的两种核心机制——**强缓存**和**协商缓存**。我们将从原理出发，通过代码示例演示如何配置，并探讨实际开发中的最佳实践。无论你是前端新手还是资深工程师，掌握缓存机制都能让你在性能优化中游刃有余。

## 什么是浏览器缓存？

浏览器缓存是指浏览器将请求过的资源（如 HTML、CSS、JavaScript、图片等）存储在本地磁盘或内存中。当用户再次请求相同资源时，浏览器可以直接从缓存中读取，而无需重新向服务器发起网络请求。这不仅可以加快页面渲染速度，还能节省带宽资源。

缓存的实现主要依赖 HTTP 响应头中的几个关键字段：`Expires`、`Cache-Control`、`Last-Modified`、`ETag`。根据缓存的验证方式，可以分为两大类：

- **强缓存**：浏览器直接使用本地缓存，不向服务器发送请求。
- **协商缓存**：浏览器向服务器发送请求，由服务器判断缓存是否有效。

下面我们逐一深入。

## 强缓存：让浏览器直接使用本地副本

强缓存的核心思想是：在缓存有效期内，浏览器直接从本地读取资源，完全不与服务器通信。这可以极大减少网络延迟，适用于不经常变化的静态资源，如 logo 图片、公共 CSS/JS 文件等。

### 强缓存的实现方式

强缓存通过两个 HTTP 响应头来控制：

1. **Expires**（HTTP/1.0 产物）
2. **Cache-Control**（HTTP/1.1 标准）

#### Expires

`Expires` 是一个绝对时间，告诉浏览器在指定时间之前可以直接使用缓存。例如：

```http
Expires: Wed, 21 Oct 2026 07:28:00 GMT
```

但 `Expires` 存在明显缺陷：它依赖客户端时间，如果用户修改了系统时间，缓存可能失效或异常。因此，现代开发中更推荐使用 `Cache-Control`。

#### Cache-Control

`Cache-Control` 是 HTTP/1.1 的规范，使用相对时间控制缓存，更加灵活。常用指令包括：

- `max-age=<seconds>`：缓存的最大有效时间，相对于请求时间。
- `public`：响应可以被任何中间节点（如 CDN）缓存。
- `private`：响应只能被浏览器缓存，不允许中间节点缓存。
- `no-cache`：强制进行协商缓存，每次请求都要验证。
- `no-store`：完全禁止缓存，每次请求都从服务器获取。

示例：设置强缓存 1 小时

```http
Cache-Control: public, max-age=3600
```

### 强缓存的代码配置

在 Node.js 中使用 Express 框架配置强缓存：

```javascript
const express = require('express');
const app = express();

app.get('/static/style.css', (req, res) => {
  // 设置强缓存，有效期1小时
  res.setHeader('Cache-Control', 'public, max-age=3600');
  res.sendFile(__dirname + '/public/style.css');
});

app.listen(3000, () => {
  console.log('Server running on port 3000');
});
```

当浏览器第一次请求 `/static/style.css` 时，服务器返回资源并附带 `Cache-Control: public, max-age=3600`。在接下来的 3600 秒内，浏览器再次请求该资源时，会直接从本地缓存读取，状态码显示为 `200 (from disk cache)` 或 `200 (from memory cache)`，网络面板中可以看到 `Size` 列为 `from cache`。

## 协商缓存：与服务器确认缓存是否有效

强缓存虽然高效，但无法处理资源更新的场景。当缓存过期后，浏览器需要向服务器确认资源是否发生了变化。如果资源未变，服务器返回 304 状态码，浏览器继续使用缓存；如果资源已变，服务器返回新资源。

这种机制称为**协商缓存**，它通过两对 HTTP 头部实现：

1. **Last-Modified / If-Modified-Since**（基于时间）
2. **ETag / If-None-Match**（基于内容指纹）

### 基于时间的协商缓存

服务器在响应头中返回 `Last-Modified`，表示资源的最后修改时间。浏览器后续请求时，在请求头中携带 `If-Modified-Since`，值为上次收到的 `Last-Modified`。服务器比较时间，如果资源未修改，返回 304；否则返回 200 和新资源。

示例响应头：

```http
Last-Modified: Tue, 01 Jun 2026 10:15:30 GMT
```

示例请求头：

```http
If-Modified-Since: Tue, 01 Jun 2026 10:15:30 GMT
```

### 基于内容的协商缓存

`Last-Modified` 存在精度问题（秒级）和内容未修改但时间变化的情况（如文件被重写）。`ETag` 解决了这些问题，它通过文件内容的哈希值（如 MD5）来判断资源是否变化。

服务器返回 `ETag`：

```http
ETag: "5d8c72a5edda8a6f3b3f5c1a2b3c4d5e"
```

浏览器后续请求携带 `If-None-Match`：

```http
If-None-Match: "5d8c72a5edda8a6f3b3f5c1a2b3c4d5e"
```

如果 `ETag` 匹配，服务器返回 304；否则返回 200 和新资源。

### 协商缓存的代码配置

在 Express 中配置协商缓存：

```javascript
const express = require('express');
const fs = require('fs');
const crypto = require('crypto');
const app = express();

app.get('/static/script.js', (req, res) => {
  const filePath = __dirname + '/public/script.js';
  const fileContent = fs.readFileSync(filePath);
  
  // 计算 ETag（基于文件内容的 MD5）
  const etag = crypto.createHash('md5').update(fileContent).digest('hex');
  
  // 检查 If-None-Match
  if (req.headers['if-none-match'] === etag) {
    res.status(304).end();
    return;
  }
  
  // 设置 Last-Modified 和 ETag
  const stats = fs.statSync(filePath);
  res.setHeader('Last-Modified', stats.mtime.toUTCString());
  res.setHeader('ETag', etag);
  res.sendFile(filePath);
});

app.listen(3000);
```

当浏览器请求 `/static/script.js` 时，服务器返回资源并附带 `Last-Modified` 和 `ETag`。下次请求时，浏览器自动在请求头中添加 `If-Modified-Since` 和 `If-None-Match`，服务器根据这些值决定返回 304 还是 200。

## 强缓存与协商缓存的协作流程

在实际应用中，强缓存和协商缓存通常配合使用。浏览器会先检查强缓存是否有效：

1. 如果强缓存未过期（`Cache-Control: max-age` 或 `Expires` 有效），直接使用本地缓存，不发送请求。
2. 如果强缓存已过期，浏览器发起请求，并在请求头中携带协商缓存的相关字段（`If-Modified-Since` 或 `If-None-Match`）。
3. 服务器根据这些字段判断资源是否变化：
   - 未变化：返回 304，浏览器继续使用缓存。
   - 已变化：返回 200 和新资源，同时更新缓存头部。

这种分层机制既保证了性能（强缓存减少请求），又保证了数据准确性（协商缓存验证更新）。

## 实际开发中的缓存策略最佳实践

### 静态资源：强缓存 + 版本号

对于不经常变化的静态资源（如 CSS、JS、图片），推荐使用强缓存，并配合文件名哈希或版本号实现更新。

例如：

```html
<link rel="stylesheet" href="/static/style.a1b2c3d4.css">
```

服务器配置强缓存（如一年）：

```http
Cache-Control: public, max-age=31536000, immutable
```

当文件内容变化时，构建工具自动生成新的哈希文件名，浏览器视为全新资源，强制请求最新版本。这种方式既享受了强缓存的性能优势，又避免了缓存更新问题。

### HTML 文件：协商缓存

HTML 文件通常需要实时更新，因此适合使用协商缓存。配置 `Cache-Control: no-cache`，每次请求都验证资源是否变化。

```http
Cache-Control: no-cache
ETag: "abc123"
Last-Modified: Tue, 01 Jun 2026 10:15:30 GMT
```

### 敏感数据：禁止缓存

对于涉及用户隐私或实时性要求极高的数据（如 API 响应），应使用 `no-store` 完全禁止缓存。

```http
Cache-Control: no-store
```

### 使用 Nginx 配置缓存

在生产环境中，Nginx 是常用的反向代理服务器，可以轻松配置缓存策略：

```nginx
location /static/ {
    expires 1y;
    add_header Cache-Control "public, immutable";
}

location / {
    add_header Cache-Control "no-cache";
}
```

## 总结

浏览器缓存机制是前端性能优化的基石。通过合理配置强缓存和协商缓存，我们可以大幅减少不必要的网络请求，提升页面加载速度，降低服务器压力。

核心要点回顾：
- **强缓存**：通过 `Cache-Control` 和 `Expires` 控制，在有效期内直接使用本地缓存，不发送请求。
- **协商缓存**：通过 `Last-Modified/If-Modified-Since` 和 `ETag/If-None-Match` 实现，向服务器验证资源是否更新。
- **实践建议**：静态资源使用强缓存 + 文件名哈希；HTML 文件使用协商缓存；敏感数据禁止缓存。

掌握这些知识后，你将能够根据业务场景灵活设计缓存策略，为用户提供更流畅的浏览体验。缓存虽小，却是性能优化中“四两拨千斤”的关键技术。希望本文能帮助你彻底理解浏览器缓存机制，并在实际项目中得心应手。
