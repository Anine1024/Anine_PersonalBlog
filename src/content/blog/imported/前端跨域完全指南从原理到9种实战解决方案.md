## 引言

在前端开发中，跨域问题几乎是每位开发者都绕不开的“拦路虎”。无论是调用第三方API，还是与不同子域名的后端服务进行交互，浏览器那看似“严苛”的同源策略总会跳出来阻止你的请求。很多新手开发者第一次遇到跨域错误时，往往一头雾水，甚至以为是后端接口写错了。

实际上，跨域问题的本质是**浏览器的安全机制**与**业务需求**之间的矛盾。理解跨域，不仅能帮你快速定位问题，还能让你在设计系统架构时做出更合理的选择。本文将从跨域的根源——同源策略讲起，然后逐一剖析9种主流跨域解决方案的原理、适用场景及代码实现，帮你建立完整的跨域知识体系。

## 一、跨域的本质：同源策略

### 1.1 什么是同源？

“同源”指的是三个要素完全相同：**协议（Protocol）**、**域名（Host）**、**端口（Port）**。只要任意一个不同，就构成了跨域。

| 当前页面URL | 目标URL | 是否同源 | 原因 |
|------------|---------|---------|------|
| `https://a.com/page` | `https://a.com/api` | ✅ 是 | 协议、域名、端口均相同 |
| `https://a.com/page` | `http://a.com/api` | ❌ 否 | 协议不同（https vs http） |
| `https://a.com/page` | `https://b.com/api` | ❌ 否 | 域名不同 |
| `https://a.com:8080/page` | `https://a.com:3000/api` | ❌ 否 | 端口不同 |

### 1.2 同源策略限制了什么？

同源策略主要限制以下三种行为：

1. **DOM访问限制**：不同源的页面之间无法通过JavaScript操作彼此的DOM。
2. **数据读取限制**：无法通过`XMLHttpRequest`或`Fetch`读取不同源资源返回的数据。
3. **网络请求限制**：虽然可以发送请求（如图片、脚本），但浏览器会拦截响应内容。

### 1.3 为什么要有同源策略？

想象一下，如果没有同源策略，你登录了银行网站后，又打开了一个恶意网站。这个恶意网站可以轻松地通过脚本向你的银行账户发起转账请求，因为浏览器会“自动携带”你的登录Cookie。同源策略正是为了防止这种**跨站请求伪造（CSRF）**和**跨站脚本攻击（XSS）**而存在的。

> 关键点：同源策略是**浏览器**的行为，不是服务器的行为。服务器依然可以收到请求，但浏览器会拒绝让前端代码读取响应数据。

## 二、跨域解决方案全景图

解决跨域的核心思路无非两种：
- **绕过同源策略**：利用某些不受同源限制的标签或API（如`<img>`、`<script>`、`postMessage`）。
- **让服务器配合**：通过CORS、代理服务器等方式，让浏览器“认可”这次跨域请求。

下面我们逐一介绍9种常用方案。

## 三、方案一：JSONP（JSON with Padding）

### 3.1 原理

`<script>`标签的`src`属性不受同源策略限制。JSONP利用这一点，通过动态创建`<script>`标签，将请求参数拼接在URL中，服务器返回一段JavaScript代码（通常是函数调用），前端提前定义好回调函数来接收数据。

### 3.2 代码示例

```javascript
// 前端代码
function jsonp(url, params, callbackName) {
  return new Promise((resolve, reject) => {
    // 1. 动态创建 script 标签
    const script = document.createElement('script');
    
    // 2. 将回调函数挂载到 window 上
    window[callbackName] = function(data) {
      resolve(data);
      // 清理
      document.body.removeChild(script);
      delete window[callbackName];
    };
    
    // 3. 拼接参数
    const queryString = Object.entries(params)
      .map(([key, value]) => `${encodeURIComponent(key)}=${encodeURIComponent(value)}`)
      .join('&');
    
    script.src = `${url}?${queryString}&callback=${callbackName}`;
    
    script.onerror = () => {
      reject(new Error('JSONP request failed'));
      document.body.removeChild(script);
    };
    
    document.body.appendChild(script);
  });
}

// 使用示例
jsonp('https://api.example.com/user', { id: 123 }, 'handleUser')
  .then(data => console.log('用户数据:', data))
  .catch(err => console.error(err));
```

### 3.3 优缺点

**优点**：
- 兼容性极好，支持所有浏览器（包括IE6）
- 实现简单

**缺点**：
- **只支持GET请求**，因为script标签只能发起GET请求
- 无法处理请求失败（虽然可以监听onerror，但无法获取HTTP状态码）
- 存在安全风险（如果服务器被攻破，返回恶意代码，前端会直接执行）

### 3.4 适用场景

- 需要兼容老旧浏览器的场景
- 仅需要GET请求获取数据的场景（如百度搜索联想词）

## 四、方案二：CORS（跨域资源共享）

CORS是目前最主流的跨域解决方案，它需要**服务器端配合**设置响应头。

### 4.1 原理

浏览器在发送跨域请求时，会自动添加`Origin`请求头。服务器通过设置`Access-Control-Allow-Origin`响应头，告诉浏览器“我允许这个源访问我”。浏览器检查响应头后，如果允许，则把数据交给前端代码；否则触发错误。

### 4.2 简单请求 vs 预检请求

CORS将请求分为两类：

**简单请求**（满足以下所有条件）：
- 方法为GET、HEAD或POST
- Content-Type为`application/x-www-form-urlencoded`、`multipart/form-data`或`text/plain`
- 没有自定义请求头

**预检请求**（不满足简单请求的）：
- 浏览器会先发一个`OPTIONS`请求，询问服务器是否允许
- 服务器返回允许的方法和头信息后，再发送真实请求

### 4.3 代码示例

**前端代码（无需特殊处理）**：
```javascript
fetch('https://api.example.com/data', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',  // 这会导致预检请求
    'Authorization': 'Bearer xxx'
  },
  body: JSON.stringify({ name: 'Alice' })
})
.then(response => response.json())
.then(data => console.log(data));
```

**服务端（Node.js + Express）**：
```javascript
const express = require('express');
const app = express();

// 全局CORS中间件
app.use((req, res, next) => {
  // 允许特定源（生产环境不要用 *）
  res.setHeader('Access-Control-Allow-Origin', 'https://frontend.com');
  
  // 允许携带Cookie（需要设置具体Origin，不能是 *）
  res.setHeader('Access-Control-Allow-Credentials', 'true');
  
  // 允许的请求方法
  res.setHeader('Access-Control-Allow-Methods', 'GET, POST, PUT, DELETE, OPTIONS');
  
  // 允许的请求头
  res.setHeader('Access-Control-Allow-Headers', 'Content-Type, Authorization');
  
  // 预检请求的有效期（秒），减少OPTIONS请求次数
  res.setHeader('Access-Control-Max-Age', '86400');
  
  // 处理预检请求
  if (req.method === 'OPTIONS') {
    return res.sendStatus(204);
  }
  
  next();
});

app.post('/data', (req, res) => {
  res.json({ message: 'CORS request successful!' });
});

app.listen(3000);
```

### 4.4 常见踩坑点

1. **`Access-Control-Allow-Origin: *` 不能与 `credentials: 'include'` 一起使用**。如果需要携带Cookie，必须指定具体的源。
2. **预检请求的OPTIONS方法**需要服务器正确处理，否则真实请求不会发出。
3. **前端需要设置`credentials: 'include'`** 才能发送Cookie。

```javascript
// 前端发送带Cookie的请求
fetch('https://api.example.com/user', {
  credentials: 'include'  // 或 'same-origin'
});
```

## 五、方案三：代理服务器

当无法修改后端代码时，代理服务器是“曲线救国”的最佳方案。

### 5.1 原理

同源策略只存在于浏览器中。我们可以搭建一个与前端同源的**中间代理服务器**，前端请求发给代理，代理转发给目标服务器，再将响应返回给前端。由于代理与前端同源，浏览器不会拦截。

### 5.2 开发环境：Webpack Dev Server

```javascript
// vue.config.js 或 webpack.config.js
module.exports = {
  devServer: {
    proxy: {
      '/api': {
        target: 'https://api.example.com',  // 目标服务器
        changeOrigin: true,                 // 修改请求头中的Host
        pathRewrite: { '^/api': '' },       // 可选：重写路径
        // 如果需要携带Cookie
        cookieDomainRewrite: 'localhost'
      }
    }
  }
};
```

前端请求时：
```javascript
// 直接请求同源的 /api 路径，代理会自动转发
fetch('/api/user')
  .then(res => res.json())
  .then(data => console.log(data));
```

### 5.3 生产环境：Nginx反向代理

```nginx
server {
    listen 80;
    server_name frontend.com;

    # 前端静态文件
    location / {
        root /var/www/frontend;
        index index.html;
    }

    # API代理
    location /api/ {
        proxy_pass https://api.example.com/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        
        # 处理Cookie
        proxy_cookie_domain api.example.com frontend.com;
        
        # 跨域头（如果需要）
        add_header Access-Control-Allow-Origin *;
    }
}
```

### 5.4 Node.js中间层代理

```javascript
const express = require('express');
const { createProxyMiddleware } = require('http-proxy-middleware');

const app = express();

// 静态文件服务
app.use(express.static('public'));

// 代理配置
app.use('/api', createProxyMiddleware({
  target: 'https://api.example.com',
  changeOrigin: true,
  onProxyReq: (proxyReq, req, res) => {
    // 可以在代理时修改请求头
    proxyReq.setHeader('X-Proxy-By', 'Node');
  }
}));

app.listen(3000);
```

## 六、方案四：postMessage（跨文档通信）

### 6.1 原理

`window.postMessage`允许不同源的窗口（如iframe、弹出窗口）之间进行安全通信。发送方调用`postMessage`发送数据，接收方监听`message`事件。

### 6.2 代码示例

**父页面 (https://parent.com)**：
```html
<iframe id="childFrame" src="https://child.com/page.html"></iframe>

<script>
  const iframe = document.getElementById('childFrame');
  
  // 发送消息给子页面
  iframe.contentWindow.postMessage(
    { type: 'GET_USER', id: 123 },
    'https://child.com'  // 目标源，严格限制
  );
  
  // 监听子页面的回复
  window.addEventListener('message', (event) => {
    // 验证消息来源
    if (event.origin !== 'https://child.com') return;
    
    console.log('收到子页面数据:', event.data);
  });
</script>
```

**子页面 (https://child.com/page.html)**：
```javascript
// 监听父页面消息
window.addEventListener('message', (event) => {
  // 验证来源
  if (event.origin !== 'https://parent.com') return;
  
  const data = event.data;
  if (data.type === 'GET_USER') {
    // 回复父页面
    event.source.postMessage(
      { name: 'Alice', age: 25 },
      event.origin
    );
  }
});
```

### 6.3 安全注意事项

- **永远验证`event.origin`**，防止恶意页面伪造消息。
- **指定精确的目标源**，不要使用`*`。
- 避免在消息中传递敏感信息（如密码、Token）。

## 七、方案五：WebSocket

WebSocket是一种全双工通信协议，它**不受同源策略限制**。浏览器在发起WebSocket连接时，会通过`Origin`头告知服务器，服务器可以选择接受或拒绝。

### 7.1 代码示例

```javascript
// 前端代码
const socket = new WebSocket('wss://api.example.com/ws');

socket.onopen = () => {
  socket.send(JSON.stringify({ type: 'subscribe', channel: 'news' }));
};

socket.onmessage = (event) => {
  const data = JSON.parse(event.data);
  console.log('收到推送:', data);
};

socket.onerror = (err) => {
  console.error('WebSocket error:', err);
};
```

### 7.2 适用场景

- 需要实时双向通信的应用（如聊天、股票行情、协作编辑）
- 对延迟要求高的场景

## 八、方案六：document.domain + iframe

### 8.1 原理

如果两个页面属于**同一主域名但不同子域名**（如`a.example.com`和`b.example.com`），可以通过设置`document.domain`为相同的父域名来实现同源。

### 8.2 代码示例

**页面A (http://a.example.com/pageA.html)**：
```javascript
document.domain = 'example.com';

const iframe = document.createElement('iframe');
iframe.src = 'http://b.example.com/pageB.html';
iframe.onload = () => {
  // 现在可以跨域访问了
  console.log(iframe.contentDocument.body.innerHTML);
};
document.body.appendChild(iframe);
```

**页面B (http://b.example.com/pageB.html)**：
```javascript
document.domain = 'example.com';
// 现在可以与页面A通信
```

### 8.3 局限性

- 仅适用于**同一主域名**下的不同子域名
- 存在安全风险（放宽了同源限制）
- 现代开发中较少使用，CORS是更好的替代方案

## 九、方案七：window.name + iframe

### 9.1 原理

`window.name`属性有一个特性：即使页面跳转到不同域，其值仍然保持不变。利用这个特性，可以在不同域的页面间传递数据。

### 9.2 代码示例

```javascript
// 主页面 (http://main.com)
const iframe = document.createElement('iframe');
iframe.src = 'http://target.com/data';  // 目标服务器返回数据到 window.name

iframe.onload = () => {
  // 第一次加载完成后，将 iframe 指向同源的空白页面
  iframe.src = 'http://main.com/blank.html';
  
  iframe.onload = () => {
    // 现在可以安全读取 window.name
    const data = iframe.contentWindow.name;
    console.log('获取到的数据:', data);
    document.body.removeChild(iframe);
  };
};

document.body.appendChild(iframe);
```

### 9.3 优缺点

**优点**：兼容性好，支持所有浏览器。  
**缺点**：实现复杂，需要借助空白页面做中转；仅能传递字符串数据（大小限制约2MB）。

## 十、方案八：location.hash + iframe

### 10.1 原理

通过改变iframe的`location.hash`来传递数据（URL的`#`部分改变不会导致页面刷新）。需要借助一个同源的代理页面来桥接通信。

### 10.2 代码示例

```
主页面 (http://main.com/page.html)
    ↓ 设置 hash
子页面 (http://target.com/page.html)
    ↓ 通过代理页面
代理页面 (http://main.com/proxy.html)  ← 与主页面同源
    ↓ 读取 hash
主页面获取数据
```

这个方案由于实现繁琐且现代浏览器已有更好的替代方案（如postMessage），目前已很少使用。

## 十一、方案九：Chrome插件或浏览器配置（开发调试用）

### 11.1 禁用同源策略（仅开发环境）

```bash
# Mac OS
open -n -a /Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome --args --user-data-dir="/tmp/chrome_dev" --disable-web-security

# Windows
chrome.exe --disable-web-security --user-data-dir="C:\tmp\chrome_dev"
```

### 11.2 使用CORS浏览器插件

如“Allow CORS: Access-Control-Allow-Origin”等插件可以临时添加CORS头。

> 警告：这些方法仅用于本地开发调试，**绝不能用于生产环境**，否则会带来严重的安全风险。

## 总结

| 方案 | 原理 | 是否需要后端配合 | 适用场景 |
|------|------|-----------------|---------|
| JSONP | script标签不受限 | 是（返回JS代码） | 兼容老浏览器，仅GET请求 |
| CORS | 服务器设置响应头 | 是 | **现代应用首选**，支持所有HTTP方法 |
| 代理服务器 | 同源代理转发 | 否（前端搭建代理） | 无法修改后端时 |
| postMessage | 跨窗口通信API | 否 | iframe、弹窗通信 |
| WebSocket | 协议本身不受限 | 是（支持WS） | 实时双向通信 |
| document.domain | 同主域放宽限制 | 否 | 子域名间通信（已过时） |
| window.name | 属性跨域保持 | 是（返回数据） | 兼容老浏览器（已过时） |
| location.hash | URL hash传递 | 是（需代理页） | 兼容老浏览器（已过时） |
| 浏览器配置 | 关闭安全策略 | 否 | 仅本地开发调试 |

**最佳实践建议**：
1. **首选CORS**：它是W3C标准，功能完善，支持所有HTTP方法和自定义头。
2. **开发环境用代理**：方便快捷，无需修改后端代码。
3. **生产环境用Nginx代理或CORS**：根据团队技术栈和运维能力选择。
4. **避免使用过时方案**：除非需要兼容IE8/9等上古浏览器。

跨域问题本质上是一个“安全与便利的平衡”问题。理解了同源策略的初衷和每种方案的原理，你就能在遇到跨域问题时，从容地选择最合适的解决方案。希望这篇文章能帮你彻底攻克跨域这个“拦路虎”。
